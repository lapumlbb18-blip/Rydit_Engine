Función cargar_modulo():
```rust
fn cargar_modulo(nombre: &str) -> Result<String, String> {
    // 1. Intentar archivo local (modules/*.rydit)
    let ruta_local = format!("modules/{}.rydit", nombre);
    if std::path::Path::new(&ruta_local).exists() {
        std::fs::read_to_string(&ruta_local)
            .map_err(|e| format!("Error leyendo '{}': {}", nombre, e))
    } else {
        // 2. Fallback a embebido
        match nombre {
            "math" => Ok(MATH_MODULE.to_string()),
            "arrays" | "listas" => Ok(ARRAYS_MODULE.to_string()),
            "strings" | "cadenas" => Ok(STRINGS_MODULE.to_string()),
            // ... más módulos
            _ => Err(format!("Módulo '{}' no encontrado", nombre)),
        }
    }
}
```

Aliases soportados:
- arrays o listas
- strings o cadenas
- random o aleatorio
- time o tiempo

Impacto:
- Aumento binario: ~15 KB (8 módulos .rydit)
- UX: Usuario hace import math sin archivos
- Flexibilidad: Usuario avanzado puede override

C. FIX TERMUX-X11 AUTOMÁTICO (~35 líneas Rust)
----------------------------------------------
Archivo: crates/rydit-rs/src/main.rs

Función configurar_entorno_termux():
```rust
fn configurar_entorno_termux() {
    // Detectar si estamos en Termux
    let es_termux = std::env::var("TERMUX_VERSION").is_ok() || 
                    std::path::Path::new("/data/data/com.termux").exists();
    
    if es_termux {
        println!("[RYDIT] Termux detectado - Configurando entorno gráfico...");
        
        // Configurar DISPLAY si no está establecido
        if std::env::var("DISPLAY").is_err() {
            std::env::set_var("DISPLAY", ":0");
            println!("[RYDIT] DISPLAY=:0 configurado automáticamente");
        }
        
        // Configurar driver zink si no está establecido
        if std::env::var("MESA_LOADER_DRIVER_OVERRIDE").is_err() {
            std::env::set_var("MESA_LOADER_DRIVER_OVERRIDE", "zink");
            println!("[RYDIT] zink GPU driver configurado automáticamente");
        }
        
        // Configurar DRI3 si no está establecido
        if std::env::var("DRI3").is_err() {
            std::env::set_var("DRI3", "1");
            println!("[RYDIT] DRI3=1 configurado automáticamente");
        }
        
        println!("[RYDIT] ✅ Entorno gráfico listo para Termux-X11");
    }
}
```

Llamado en main():
```rust
fn main() {
    configurar_entorno_termux();  // ← NUEVO
    let args: Vec<String> = env::args().collect();
    // ... resto del código
}
```

Detección:
- TERMUX_VERSION env var (oficial de Termux)
- Path /data/data/com.termux (existe en Termux)

Variables configuradas:
- DISPLAY=:0 - Conecta a Termux-X11
- MESA_LOADER_DRIVER_OVERRIDE=zink - GPU sobre Vulkan
- DRI3=1 - Renderizado directo

                    5. ERRORES ENCONTRADOS Y SOLUCIONES
================================================================================

ERROR 1: Sound/Music structs sin campo 'id'
-------------------------------------------
Archivo: crates/rydit-gfx/src/lib.rs
Ubicación: AudioSystem::load_sound(), load_music()

Código problemático:
```rust
if sound.id != 0 {  // ❌ ERROR: no existe campo 'id'
```

Solución:
```rust
if !sound.stream.buffer.is_null() || sound.frameCount > 0 {  // ✅
```

Explicación:
- Sound y Music en raylib FFI no tienen campo 'id'
- Usamos stream.buffer.is_null() para verificar validez
- Alternativa: frameCount > 0

Tiempo de solución: ~10 minutos


ERROR 2: FFI requiere desreferenciación
----------------------------------------
Archivo: crates/rydit-gfx/src/lib.rs
Ubicación: PlaySound, StopSound, etc.

Código problemático:
```rust
raylib::ffi::PlaySound(sound);  // ❌ ERROR: tipo mismatch
```

Solución:
```rust
raylib::ffi::PlaySound(*sound);  // ✅ desreferenciar
```

Explicación:
- FFI functions esperan Sound por valor, no por referencia
- HashMap::get() retorna &Sound
- Necesario desreferenciar con *

Tiempo de solución: ~15 minutos


ERROR 3: HashMap import duplicado
---------------------------------
Archivo: crates/rydit-gfx/src/lib.rs

Código problemático:
```rust
use std::collections::HashMap;  // Línea 48

use std::collections::HashMap;  // Línea 808, ❌ DUPLICADO
```

Solución:
Eliminar import duplicado en línea 808

Explicación:

- Assets struct usaba segundo import innecesario
- Rust no permite imports duplicados en mismo scope

Tiempo de solución: ~2 minutos

DECISIÓN 1: Bindings - MANTENER DOBLE NOMBRE
---------------------------------------------
Opción: ¿Refactorizar __audio_play || audio::play?

Decisión: MANTENER para v0.6.0, refactor en v0.7.0

Razones:
- Funciona perfectamente actualmente
- Refactor requiere ~50 ubicaciones cambiadas
- Riesgo de introducir bugs en recta final
- Overhead es mínimo (solo un OR en if)

Código actual:
```rust
if (name == "__audio_play" || name == "audio::play") && args.len() == 1 {
    // ...
}
```

Futuro (v0.7.0):
```rust
macro_rules! bind {
    ($name:expr, $internal:expr, $args:expr, $body:block) => { ... }
}
```


DECISIÓN 2: Stdlib - EMBEBER HÍBRIDO
------------------------------------
Opción: ¿Archivos externos o embebidos?

Decisión: EMBEBER con fallback a archivos locales

Razones:
- Mejor UX para principiantes (sin gestión de archivos)
- Usuarios avanzados pueden override con archivos locales
- Aumento de binario mínimo (~15 KB)
- Compensado por strip (~100 KB menos)

Implementación:
```rust
fn cargar_modulo(nombre: &str) -> Result<String, String> {
    // 1. Intentar archivo local
    if Path::new(&ruta_local).exists() {
        return Ok(fs::read_to_string(&ruta_local)?);
    }
    // 2. Fallback a embebido
    match nombre {
        "math" => Ok(MATH_MODULE.to_string()),
        // ...
    }
}
```

Razones:

                        9. LECCIONES APRENDIDAS
================================================================================

LECCIÓN 1: API DE SESIÓN - IMPORTANTE VERIFICAR CIERRE
-------------------------------------------------------
Problema: API se cortó durante implementación de configurar_entorno_termux()
Resultado: Función definida pero no llamada
Solución: Siempre verificar que código se insertó completo antes de continuar

Lección para futuro:
- Pedir confirmación de API después de edits largos
- Verificar con grep que funciones se llaman
- Testear inmediatamente después de implementar


LECCIÓN 2: FFI REQUIERE ATENCIÓN EXTRA A TIPOS
----------------------------------------------
Problema: Sound/Music sin campo 'id', requiere desreferenciación
Resultado: 2 errores de compilación
Solución: Leer documentación FFI cuidadosamente, usar * para desreferenciar

Lección para futuro:
- Verificar tipos FFI antes de implementar
- Usar rust-analyzer para ver tipos exactos
- Testear FFI en aislamiento primero


LECCIÓN 3: REFACTOR EN RECTA FINAL = RIESGO INNECESARIO
-------------------------------------------------------
Problema: Tentación de refactorizar bindings (__audio_play || audio::play)
Resultado: Decidido mantener para v0.7.0
Solución: Si funciona, no tocar hasta después de release

Lección para futuro:
- Lista de "refactor para vNext" separada de "must have"
- Resistir perfeccionismo en recta final
- Documentar deuda técnica para futuro


LECCIÓN 4: STDLIB EMBEBIDO = MEJOR UX SIN SACRIFICAR FLEXIBILIDAD
-----------------------------------------------------------------
Problema: Usuarios necesitan archivos externos para imports
Resultado: 8 módulos embebidos con fallback local
Solución: include_str!() + función cargar_modulo() híbrida

Lección para futuro:
- Embeber defaults, permitir override
- 15 KB es aceptable por mejora en UX
- Documentar claramente cómo hacer override


LECCIÓN 5: DETECCIÓN AUTOMÁTICA > SCRIPT EXTERNO
------------------------------------------------
Problema: Usuarios olvidan variables de entorno
Resultado: Detección en Rust con mensajes informativos
Solución: configurar_entorno_termux() al inicio de main()

Lección para futuro:
- Automatizar configuración siempre que posible
- Mensajes claros de qué se configuró
- Permitir override con variables manuales


LECCIÓN 6: DEMOS QUE FUNCIONAN = MEJOR TEST QUE CUALQUIER COSA
--------------------------------------------------------------
Problema: ¿Cómo saber si partículas funciona?
Resultado: Usuario reportó "funcionó de maravilla"
Solución: Demo binary independiente que se puede ejecutar

Lección para futuro:
- Crear demos ejecutables para cada feature mayor
- Testear en hardware real (no solo emulación)
- Feedback de usuario > tests automáticos para UX
