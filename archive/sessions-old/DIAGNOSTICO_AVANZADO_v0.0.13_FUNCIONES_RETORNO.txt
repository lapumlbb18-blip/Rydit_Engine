PROBLEMA PRINCIPAL: FUNCIONES SIN RETORNO DE VALORES
-----------------------------------------------------

CUÁNDO OCURRIÓ:
---------------
Al analizar el diagnóstico v0.0.12, se identificó que:
- Las funciones (`rytmo`) solo podían hacer `voz` (print)
- No podían retornar valores para usar en expresiones
- `return` solo imprimía, no retornaba realmente

SÍNTOMAS:
---------
1. `dark.slot x = mi_funcion(5)` → Error
2. Funciones solo como statements, no en expresiones
3. Imposible componer funciones: `sumar(multiplicar(2, 3), 4)`

CAUSA RAÍZ TÉCNICA:
-------------------
1. `Stmt::Return` solo hacía `executor.voz(&val)` (imprimir)
2. `ejecutar_stmt` retornaba `Option<bool>` (solo para break)
3. No había mecanismo para propagar valores de retorno
4. `evaluar_expr` tenía `&Executor` en vez de `&mut Executor`

================================================================================
                         ⚡ REACCIÓN VERSÁTIL
================================================================================

FASE 1: DIAGNÓSTICO (10 minutos)
--------------------------------
- Identificado problema en Stmt::Return
- Analizado flujo de ejecución de funciones
- Detectado que return no propagaba valor

FASE 2: ANÁLISIS (15 minutos)
-----------------------------
- `ejecutar_stmt` necesita retornar `(Option<bool>, Option<Valor>)`
- `evaluar_expr` necesita `&mut Executor` para funciones de usuario
- Pero `evaluar_expr` se llama desde contextos con `&Executor`
- **Problema de diseño fundamental**: separación entre statements y expresiones

FASE 3: SOLUCIÓN PARCIAL (45 minutos)
-------------------------------------
- Cambiado firma de `ejecutar_stmt` a `(Option<bool>, Option<Valor>)`
- Implementado retorno en Stmt::Return
- Actualizado Stmt::Call para capturar retorno
- Actualizado `ejecutar_programa` para imprimir retornos globales
- **Limitación**: funciones de usuario NO pueden usarse en expresiones

FASE 4: PREVENCIÓN (15 minutos)
-------------------------------
- Test manual: test_funciones_retorno.rydit
- Tests automáticos actualizados (60 pasando)
- Documentación de limitación en código

TIEMPO TOTAL: ~85 minutos

CAMBIO #1: Firma de ejecutar_stmt
---------------------------------
ARCHIVO: crates/rydit-rs/src/main.rs

ANTES:
```rust
fn ejecutar_stmt(stmt: &Stmt, executor: &mut Executor, funcs: &mut HashMap<...>) -> Option<bool>
```

DESPUÉS:
```rust
fn ejecutar_stmt(stmt: &Stmt, executor: &mut Executor, funcs: &mut HashMap<...>) -> (Option<bool>, Option<Valor>)
```

IMPACTO:
- ✅ Ahora puede retornar valor de retorno de funciones
- ✅ Propagación de break y return separada
- - Requiere actualizar 46 llamadas a evaluar_expr
- - Requiere actualizar todos los match en loops

CAMBIO #2: Stmt::Return
-----------------------
ANTES:
```rust
Stmt::Return(expr) => {
    if let Some(e) = expr {
        let val = evaluar_expr(e, executor);
        executor.voz(&val);  // Solo imprime
    }
}
```

DESPUÉS:
```rust
Stmt::Return(expr) => {
    if let Some(e) = expr {
        let val = evaluar_expr(e, executor, funcs);
        return (None, Some(val));  // Retornar valor
    } else {
        return (None, Some(Valor::Vacio));
    }
}
```

IMPACTO:
- ✅ Return ahora propaga valor al llamador
- ✅ Funciones pueden retornar números, textos, bools
- ⚠️ Return en nivel global imprime valor (comportamiento observado)

CAMBIO #3: Stmt::Call (funciones de usuario)
--------------------------------------------
AGREGADO: Captura de valor de retorno

```rust
// Ejecutar body de la función y capturar retorno
let body_clone = body.clone();
let mut return_value: Option<Valor> = None;
for s in &body_clone {
    match ejecutar_stmt(s, executor, funcs) {
        (Some(true), _) => { /* break */ }
        (_, Some(val)) => {
            return_value = Some(val);
            break;  // Salir del loop, hay retorno
        }
        _ => {}
    }
}
executor.pop_scope();

// Si hubo retorno, propagarlo
if let Some(val) = return_value {
    return (None, Some(val));
}
```

IMPACTO:
- ✅ Funciones de usuario retornan valores
- ✅ Scope se limpia correctamente
- ⚠️ Solo funciona cuando función es statement, no en expresiones

DEBILIDADES DEL PROYECTO:
-------------------------

1. **FUNCIONES NO SON EXPRESIONES** ⭐⭐
   - `dark.slot x = funcion(5)` NO funciona
   - `funcion1(funcion2(3))` NO funciona
   - Solo statements: `funcion(5)` (imprime retorno)
   
   IMPACTO: Alto - limita expresividad severamente
   
   CAUSA RAÍZ: 
   - evaluar_expr tiene `&Executor` (inmutable)
   - Ejecutar función necesita `&mut Executor` (para scopes)
   - Conflicto de borrowing en Rust
   
   SOLUCIÓN FUTURA (v0.2.0):
   - Opción A: Cambiar evaluar_expr a `&mut Executor`
   - Opción B: Separar evaluación de funciones puras
   - Opción C: Usar RefCell para mutabilidad interior

2. **RETORNO EN NIVEL GLOBAL IMPRIME** ⭐⭐
   - Si retornas en nivel global, se imprime
   - Comportamiento inesperado
   - Debería ser error de sintaxis
   
   IMPACTO: Bajo - raro en uso normal
   
   SOLUCIÓN: Validar return solo en contexto de función

3. **BREAK/RETURN EN IF ANIDADO** ⭐⭐⭐
   - Return dentro de `onif` en función puede no propagarse bien
   - Testing limitado con estructuras anidadas
   
   IMPACTO: Medio - afecta casos de uso reales
   
   SOLUCIÓN: Más tests con estructuras anidadas

4. **COMPLEJIDAD DE FIRMAS** ⭐⭐⭐
   - `(Option<bool>, Option<Valor>)` es confuso
   - Difícil de mantener
   - Podría simplificarse con enum
   
   SOLUCIÓN:
   ```rust
   enum ControlFlow {
       Continue,
       Break,
       Return(Valor)
   }
   ```

================================================================================
                         📋 LECCIONES APRENDIDAS
================================================================================

1. **CAMBIOS DE FIRMAS SON CAROS**
   Cambiar `Option<bool>` a `(Option<bool>, Option<Valor>)` requirió:
   - Actualizar 46 llamadas
   - Fixear 2 tests
   - Revisar todo el código
   
   APLICACIÓN FUTURA: Diseñar firmas correctamente desde el inicio.
   Considerar enums para múltiples retornos.

2. **&Executor VS &mut Executor**
   El conflicto entre inmutable/mutable es fundamental en Rust.
   evaluar_expr necesita mutabilidad para funciones.
   
   APLICACIÓN FUTURA: Usar &mut desde el inicio O usar RefCell
   para mutabilidad interior.

3. **TESTS AUTOMÁTICOS SALVAN VIDAS**
   Los 60 tests catchearon errores de firma inmediatamente.
   Sin tests, el refactor habría sido mucho más lento.
   
   APLICACIÓN FUTURA: Mantener cultura de tests, agregar más.

4. **SEPARACIÓN STATEMENTS/EXPRESIONES ES FUNDAMENTAL**
   Statements ejecutan, expresiones evalúan.
   Mezclarlos requiere diseño cuidadoso.
   
   APLICACIÓN FUTURA: Documentar esta separación claramente.

5. **SED SCRIPTS PARA REFACTOR MASIVO**
   `sed -i 's/evaluar_expr(\([^,)]*\), executor)/evaluar_expr(\1, executor, funcs)/g'`
   Actualizó 46 líneas en segundos.
   
   APLICACIÓN FUTURA: Usar herramientas de refactor cuando aplique.
