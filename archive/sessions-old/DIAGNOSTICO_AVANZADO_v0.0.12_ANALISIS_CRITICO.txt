CAMBIO #1: Soporte de Paréntesis en Parser
------------------------------------------
ARCHIVO: crates/lizer/src/lib.rs

PROBLEMA:
parse_primary() no reconocía paréntesis para agrupación de expresiones.

SOLUCIÓN:
```rust
// Paréntesis para agrupación: ( expr )
if matches!(self.tokens[self.pos], Token::ParentIzq) {
    self.pos += 1; // consumir (
    let expr = self.parse_expression()?;
    // Consumir )
    if self.pos < self.tokens.len() && matches!(self.tokens[self.pos], Token::ParentDer) {
        self.pos += 1;
        return Ok(expr);
    } else {
        return Err(RyDitError {
            kind: ErrorKind::SyntaxError,
            message: "Se esperaba ')' para cerrar paréntesis".to_string(),
            line: 1,
            column: self.pos,
            source: "(".to_string(),
        });
    }
}
```

IMPACTO:
- ✅ (2 + 3) * 4 = 20 (correcto)
- ✅ ((2 + 3) * (4 - 1)) = 15 (anidados)
- ✅ Precedencia alterable explícitamente

CAMBIO #2: Funciones Builtin Aritméticas
----------------------------------------
ARCHIVO: crates/rydit-rs/src/main.rs

PROBLEMA:
No había funciones builtin para operaciones aritméticas en expresiones.
Los usuarios no podían hacer: dark.slot x = sumar(1, 2, 3) * 2

SOLUCIÓN:
```rust
// Funciones aritméticas builtin en evaluar_expr()
if name == "sumar" && args.len() >= 2 {
    let mut suma = 0.0;
    for arg in args {
        if let Valor::Num(n) = evaluar_expr(arg, executor) {
            suma += n;
        } else {
            return Valor::Error("sumar() requiere números".to_string());
        }
    }
    return Valor::Num(suma);
}

// ... restar, multiplicar, dividir con manejo de errores
```

IMPACTO:
- ✅ sumar(1, 2, 3, 4, 5) = 15
- ✅ restar(10, 3) = 7
- ✅ multiplicar(2, 3, 4) = 24
- ✅ dividir(20, 4) = 5
- ✅ dividir(10, 0) = Error (manejo seguro)

DEBILIDADES DEL PROYECTO:
-------------------------

1. **FUNCIONES DE USUARIO SIN RETORNO** ⭐⭐
   - Las funciones definidas con `rytmo` no retornan valores
   - Solo pueden hacer `voz` (print)
   - No se pueden usar en expresiones: `dark.slot x = mi_funcion()`
   
   IMPACTO: Alto - limita expresividad del lenguaje
   
   SOLUCIÓN FUTURA (v0.2.0):
   - Implementar `return <expr>` que retorne valor
   - Permitir funciones en expresiones
   - Sistema de tipos para validación

2. **NO HAY SISTEMA DE MÓDULOS/IMPORT** ⭐⭐
   - Todo el código debe estar en un archivo
   - No se puede importar código de otros archivos
   - Scripts grandes son difíciles de mantener
   
   IMPACTO: Medio - afecta escalabilidad
   
   SOLUCIÓN FUTURA (v0.3.0):
   - Implementar `import "archivo.rydit"`
   - Sistema de módulos estándar
   - Librería de funciones builtin

3. **FUNCIONES BUILTIN LIMITADAS** ⭐⭐⭐
   - Solo 4 funciones aritméticas: sumar, restar, multiplicar, dividir
   - No hay pow, sqrt, modulo, abs, min, max
   - No hay funciones de string (len, substr, etc.)
   
   IMPACTO: Medio - limita operaciones matemáticas
   
   SOLUCIÓN FUTURA (v0.0.13):
   - Agregar pow(base, exp), sqrt(n), modulo(a, b)
   - Funciones de string básicas

4. **GRÁFICOS SIN SPRITES/IMÁGENES** ⭐⭐⭐
   - Solo 4 comandos draw.* (círculo, rect, línea, texto)
   - No se pueden cargar imágenes
   - No hay sistema de sprites para juegos
   
   IMPACTO: Medio - limita juegos complejos
   
   SOLUCIÓN FUTURA (v0.2.0):
   - texture.cargar("archivo.png")
   - texture.dibujar(x, y)
   - Sistema de sprites básico

5. **SONIDO NO INTEGRADO** ⭐
   - blast-core existe pero no se usa en gráficos
   - No hay comandos sound.* funcionales
   - Snake game es silencioso

================================================================================
                         📋 LECCIONES APRENDIDAS
================================================================================

1. **PARÉNTESIS SON ESENCIALES PARA EXPRESIONES COMPLEJAS**
   Sin paréntesis, no hay forma de alterar precedencia de operadores.
   Implementación simple en parse_primary() (~20 líneas).
   
   APLICACIÓN FUTURA: Siempre verificar casos edge en parser antes de asumir
   que algo "no se necesita".

2. **FUNCIONES BUILTIN VS OPERADORES**
   Operadores (+, -, *, /) son sintaxis infija.
   Funciones (sumar, restar) son sintaxis prefija y permiten ≥2 argumentos.
   Ambos son complementarios, no excluyentes.
   
   APLICACIÓN FUTURA: Cuando algo no pueda ser operador, considerar función builtin.

3. **MANEJO DE ERRORES ES CRÍTICO EN FUNCIONES**
   División por cero debe detectarse explícitamente.
   Tipos inválidos deben reportarse con mensajes claros.
   
   APLICACIÓN FUTURA: Todas las funciones builtin deben validar argumentos.

4. **TESTS MANUALES COMPLEMENTAN AUTOMÁTICOS**
   59 tests automáticos cubren código.
   20 tests manuales (test_aritmetica.rydit) cubren casos de uso.
   Ambos son necesarios para confianza total.
   
   APLICACIÓN FUTURA: Crear scripts .rydit para cada feature nueva.

5. **DIAGNÓSTICO AVANZADO CON CRÍTICA EXIGENTE ES ESENCIAL**
   Ver el proyecto con objetividad ayuda a:
   - Identificar debilidades antes de que sean críticas
   - Mantener contexto entre sesiones
   - Planificar próximas versiones realisticamente
   
   APLICACIÓN FUTURA: Crear diagnóstico avanzado en CADA sesión.
