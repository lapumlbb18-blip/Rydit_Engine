```rust
fn parse_import(&mut self) -> Result<Option<Stmt>> {
    // import <modulo> [as <alias>]
    self.pos += 1; // consumir import
    
    let module = if let Token::Ident(name) = &self.tokens[self.pos] {
        name.clone()
    } else { /* error */ };
    
    let alias = if self.pos < self.tokens.len() && matches!(self.tokens[self.pos], Token::As) {
        // Parsear alias
        Some(alias_name)
    } else {
        None
    };
    
    Ok(Some(Stmt::Import { module, alias }))
}
```

CAMBIO #3: Executor de Módulos
------------------------------
CÓDIGO AGREGADO (~80 líneas en main.rs):

```rust
Stmt::Import { module, alias } => {
    let module_path = format!("crates/modules/{}.rydit", module);
    
    if let Ok(content) = std::fs::read_to_string(&module_path) {
        // 1. Parsear módulo
        let tokens = Lizer::new(&content).scan();
        let program = Parser::new(tokens).parse()?;
        
        // 2. Recolectar funciones originales
        let mut original_funcs = Vec::new();
        for s in &program.statements {
            if let Stmt::Function { name, .. } = s {
                original_funcs.push(name.clone());
            }
        }
        
        // 3. Ejecutar módulo
        for s in &program.statements {
            ejecutar_stmt(s, executor, funcs);
        }
        
        // 4. Renombrar funciones con namespace
        let prefix = alias.unwrap_or(module);
        for orig_name in &original_funcs {
            let new_name = format!("{}::{}", prefix, orig_name);
            funcs.insert(new_name, func_data.clone());
        }
    }
}
```

IMPACTO:
- ✅ Módulos se cargan dinámicamente
- ✅ Funciones registradas con namespace
- ⚠️ Cada import re-ejecuta el módulo (ineficiente)

DEBILIDADES CRÍTICAS (EXIGENTE):
--------------------------------

1. **RE-EJECUCIÓN DE MÓDULOS** ⭐⭐
   PROBLEMA: Cada `import math` re-ejecuta todo el módulo
   IMPACTO: Performance - O(n*m) donde n=imports, m=funciones
   
   EJEMPLO:
   ```rydit
   import math      # Ejecuta 8 funciones
   import math as m # Ejecuta 8 funciones OTRA VEZ
   import math as x # Ejecuta 8 funciones OTRA VEZ
   # Total: 24 ejecuciones innecesarias
   ```
   
   SOLUCIÓN: Cache de módulos importados
   ```rust
   static mut IMPORTED_MODULES: HashSet<String> = HashSet::new();
   
   if !IMPORTED_MODULES.contains(&module) {
       // Ejecutar módulo
       IMPORTED_MODULES.insert(module.clone());
   }
   ```

2. **SIN VALIDACIÓN DE IMPORTS CÍCLICOS** ⭐
   PROBLEMA: `import a` → `import b` → `import a` = loop infinito
   
   EJEMPLO POTENCIAL:
   ```rydit
   # math.rydit
   import arrays  # Math necesita arrays
   
   # arrays.rydit
   import math  # Arrays necesita math
   # ❌ LOOP INFINITO
   ```
   
   SOLUCIÓN: Stack de imports en curso
   ```rust
   thread_local! {
       static IMPORTING: RefCell<HashSet<String>> = RefCell::new(HashSet::new());
   }
   
   if IMPORTING.with(|s| s.borrow().contains(&module)) {
       return Err("Import cíclico detectado");
   }
   ```

3. **RUTA HARDCODED** ⭐⭐
   PROBLEMA: `crates/modules/{}.rydit` está hardcodeado
   
   IMPACTO:
   - No se pueden tener módulos en otros directorios
   - No se pueden importar módulos de proyectos externos
   - Acoplamiento fuerte con estructura de directorios
   
   SOLUCIÓN: RYDIT_PATH environment variable
   ```rust
   let paths = env::var("RYDIT_PATH")
       .unwrap_or_else(|_| "crates/modules".to_string());
   
   for path in paths.split(':') {
       let module_path = format!("{}/{}.rydit", path, module);
       if std::path::Path::new(&module_path).exists() {
           // Cargar desde aquí
           break;
       }
   }
   ```

4. **FALLBACK DE NOMBRES PELIGROSO** ⭐⭐
   PROBLEMA: El fallback de `math::sumar` → `sumar` puede causar colisiones
   
   EJEMPLO:
   ```rydit
   import math
   import mi_lib
   
   dark.slot x = sumar(1, 2)  # ¿Cuál sumar? ¿math::sumar o mi_lib::sumar?
   ```
   
   SOLUCIÓN: Error explícito en caso de ambigüedad
   ```rust
   let candidates: Vec<_> = funcs.keys()
       .filter(|k| k.ends_with(&format!("::{}", name)))
       .collect();
   
   if candidates.len() > 1 {
       return Valor::Error(format!(
           "Función '{}' es ambigua. Candidatos: {:?}",
           name, candidates
       ));
   }
   ```

5. **MÓDULOS SIN ESTADO** ⭐⭐⭐
   PROBLEMA: Los módulos solo exportan funciones, no variables/constantes
   
   EJEMPLO DESEADO:
   ```rydit
   import math
   voz math::PI  # ❌ No funciona
   voz math::E   # ❌ No funciona
   ```
   
   SOLUCIÓN: Exportar también variables globales
   ```rust
   // En el módulo
   dark.slot PI = 3.14159
   dark.slot E = 2.71828
   
   rytmo sumar(a, b) { ... }
   
   // En el executor
   for (name, value) in module_globals {
       executor.guardar(&format!("{}::{}", prefix, name), value);
   }
   ```

6. **PARSE_IF ARREGLADO PERO FRÁGIL** ⭐⭐⭐
   PROBLEMA: El fix de parse_if con bloques funcionó, pero fue un parche
   
   ANÁLISIS:
   - parse_if() original no manejaba bloques { }
   - Los módulos usaban `onif x > 0 { return x }`
   - El parser fallaba silenciosamente
   - Tardé 2 horas en diagnosticar porque el error decía "columna 154"
   
   LECCIÓN: El parser debería ser más explícito con errores de sintaxis
   ```rust
   // En vez de:
   Err(RyDitError { column: self.pos, ... })  // self.pos = 154 (¿qué?)
   
   // Debería ser:
   Err(RyDitError {
       line: token_line,
       column: token_column,
       source: self.tokens[self.pos].to_string(),
       ...
   })
   ```

AMENAZAS TÉCNICAS:
------------------

1. **COMPLEJIDAD CRECIENTE**
   PROBABILIDAD: Alta
   IMPACTO: Medio
   
   El lexer tiene ~2000 líneas.
   El parser tiene ~1500 líneas.
   El executor tiene ~1200 líneas.
   
   RIESGO: Difícil de mantener para una sola persona.
   
   MITIGACIÓN:
   - Separar en crates más pequeños
   - Documentar cada función pública
   - Tests de regresión para cada bug fix

2. **PERFORMANCE DE IMPORTS**
   PROBABILIDAD: Media
   IMPACTO: Alto (para proyectos grandes)
   
   Cada import re-ejecuta el módulo completo.
   Con 10 imports y 50 funciones por módulo = 500 ejecuciones.
   
   MITIGACIÓN:
   - Cache de módulos (ver arriba)
   - Lazy loading de funciones

3. **DEUDA TÉCNICA DE ERRORES**
   PROBABILIDAD: Alta
   IMPACTO: Bajo (pero molesto)
   
   Los errores del parser usan `column: self.pos` que es el índice del token,
   no la columna real en el archivo.
   
   MITIGACIÓN:
   - Guardar línea/columna en cada Token
   - Usar esa información en errores

1. **BINDING UNIVERSAL** ⭐⭐⭐⭐⭐
   IDEA: Un solo binding que funcione para TODO tipo de función externa
   
   ACTUALMENTE:
   ```rust
   // En evaluator_expr()
   if name == "sumar" { /* 20 líneas */ }
   ```

   PROPUESTA:
   ```rust
   // Registry de bindings
   struct BindingRegistry {
       functions: HashMap<String, BindingFn>,
   }
   
   type BindingFn = fn(&[Valor]) -> Result<Valor>;
   
   // Registro
   registry.register("math::sumar", |args| {
       let a = args[0].as_num()?;
       let b = args[1].as_num()?;
       Ok(Valor::Num(a + b))
   });
   
   // Uso
   match registry.call(&name, &args) {
       Ok(result) => result,
       Err(_) => /* buscar en funciones de usuario */,
   }
   ```
   
   VENTAJAS:
   - ✅ Código más limpio (~100 líneas menos)
   - ✅ Fácil agregar nuevos bindings
   - ✅ Tests unitarios por binding
   - ✅ Bindings externos (plugins)

2. **MÓDULOS COMPILADOS** ⭐⭐⭐⭐
   IDEA: En vez de interpretar .rydit, compilar a bytecode/Rust
   
   ACTUALMENTE:
   ```rydit

   rytmo sumar(a, b) { return a + b }
   # Se parsea e interpreta cada vez
   ```
   
   PROPUESTA:
   ```rust
   // crates/modules/math.rs (compilado)
   use rydit::prelude::*;
   
   #[rydit_fn]
   fn sumar(a: f64, b: f64) -> f64 {
       a + b
   }
   
   // Se compila con el proyecto
   // Cero overhead de parsing
   ```
   
   VENTAJAS:
   - ✅ Performance 100x mejor
   - ✅ Type checking en compile-time
   - ✅ IDE support (autocomplete, go-to-definition)
   
   DESVENTAJAS:
   - ❌ Requiere recompilar para cambiar módulos
   - ❌ Más complejo para usuarios nuevos

                         📋 LECCIONES APRENDIDAS
================================================================================

1. **EL PARSER DE IF ESTABA ROTO** ⚠️
   parse_if() no manejaba bloques { } explícitos.
   Los módulos usaban `onif x > 0 { return x }`.
   El parser fallaba con error críptico "columna 154".
   
   TIEMPO DE DEBUG: 2 horas
   CAÍZA RAÍZ: Asumí que el lexer estaba bien (lo estaba).
   El problema era el parser esperando statements sueltos.
   
   LECCIÓN: Agregar tests de parser para cada tipo de statement
   con bloques { } explícitos.

2. **COLUMN EN ERRORES ES ENG AÑOSO** ⚠️
   `column: self.pos` es el índice del token, no la columna real.
   Error "columna 154" no ayuda cuando el archivo tiene 50 columnas.
   
   LECCIÓN: Guardar línea/columna en cada Token durante el scan.
   Usar esa información en errores del parser.

3. **RE-EJECUCIÓN DE MÓDULOS ES INEFICIENTE** ⚠️
   Cada `import math` re-ejecuta las 8 funciones del módulo.
   Funciona, pero es O(n*m) donde n=imports, m=funciones.
   
   LECCIÓN: Implementar cache de módulos en v0.1.2.
   HashSet de módulos ya importados.

4. **NAMESPACES PREVIENEN COLISIONES** ✅
   `math::sumar` y `mi_lib::sumar` no colisionan.
   El sistema de namespaces funciona perfectamente.
   
   LECCIÓN: Mantener este diseño. Es simple y efectivo.

5. **MÓDULOS .RYDIT SON FÁCILES DE CREAR** ✅
   Solo crear archivo en crates/modules/ y escribir funciones.
   Sin compilación, sin build steps.
   
   LECCIÓN: Mantener módulos interpretados para v0.1.x.
   Considerar módulos compilados para v0.2.0+.
