================================================================================
                         🧠 ANÁLISIS: ASYNC/AWAIT EN RYDIT
================================================================================

CONTEXTO:
---------
Rust tiene async/await nativo.
¿Cómo lo traducimos a RyDit?

OPCIÓN 1: Async Nativo (Rust-style)
-----------------------------------
```rydit
# Sintaxis estilo Rust
async rytmo descargar(url) {
    let datos = await fetch(url)
    voz datos
}

# En main:
dark.slot tarea = descargar("https://...")
await tarea
```

VENTAJAS:
- Familiar para Rustaceans
- Mapeo directo a async Rust

DESAVENTAJAS:
- Requiere runtime async (tokio, async-std)
- Complejo para principiantes
- No es "RyDit-style"

---

OPCIÓN 2: Async Implícito (Go-style goroutines)
-----------------------------------------------
```rydit
# Todo es async por defecto
rytmo tarea_lenta() {
    # Se ejecuta en background
}

# Esperar explícitamente
tarea_lenta()  # Retorna handle
wait handle    # Espera resultado
```

VENTAJAS:
- Simple
- No requiere keywords especiales

DESAVENTAJAS:
- Menos explícito
- Puede causar confusión

---

OPCIÓN 3: Async con Channels (Erlang/Elixir-style)
--------------------------------------------------
```rydit
# Processes ligeros
proc worker {
    # Código aislado
    send canal, mensaje
}

# Comunicación por channels
dark.slot canal = channel()
recv canal
```

VENTAJAS:
- Thread-safe por diseño
- Alineado con filosofía RyDit

DESAVENTAJAS:
- Diferente a Rust
- Requiere nuevo runtime

---

OPCIÓN 4: Async con Tasks (RyDit-style)
---------------------------------------
```rydit
# Task = unidad de trabajo async
task mi_tarea {
    # Código async
    pause 1000  # No bloquea
}

# Scheduler integrado
run mi_tarea
wait_all
```

VENTAJAS:
- Nativo de RyDit
- Simple y explícito
- Alineado con filosofía

DESAVENTAJAS:
- Requiere implementar scheduler
- Nuevo concepto para aprender

---

MI RECOMENDACIÓN: OPCIÓN 4 (RyDit-style Tasks)

POR QUÉ:
1. **Alineado con filosofía** - Simple, explícito, seguro
2. **No es copia de Rust** - RyDit tiene su identidad
3. **Thread-safe por diseño** - Tasks aisladas
4. **Educativo** - Enseña concurrencia real

DISEÑO PROPUESTO:
```rust
// En blast-core o crate nuevo: rydit-async

pub struct Task {
    id: usize,
    handle: JoinHandle<Value>,
}

pub struct Scheduler {
    tasks: Vec<Task>,
    channel_tx: Sender<Message>,
    channel_rx: Receiver<Message>,
}

impl Scheduler {
    pub fn spawn<F>(&mut self, f: F) -> TaskId 
    where
        F: FnOnce() -> Value + Send + 'static,
    {
        // Spawn en thread separado
    }

    pub fn wait(&mut self, task_id: TaskId) -> Value {
        // Esperar resultado
    }
}
```

SINTAXIS RYDIT:
```rydit
# Task simple
task contar {
    dark.slot i = 0
    ryda i < 10 {
        voz i
        i = i + 1
        pause 100  # ms, no bloquea
    }
}

# Ejecutar
run contar
wait_all

# Con channels
dark.slot canal = channel()

task productor {
    send canal, 42
}

task consumidor {
    dark.slot valor = recv canal
    voz valor
}

run productor, consumidor
wait_all
```

================================================================================
                         🧠 ANÁLISIS: MULTIPROCESAMIENTO
================================================================================

CONTEXTO:
---------
Rust tiene threads nativos (std::thread).
¿Cómo los exponemos en RyDit?

OPCIÓN 1: Threads Nativos (Rust-style)
--------------------------------------
```rydit
thread hilo {
    # Código
}
```

PROBLEMA:
- Threads de OS = pesados
- No es "hardware-first"
- Consume RAM en Android

---

OPCIÓN 2: Green Threads (RyDit-style)
-------------------------------------
```rydit
# Múltiples tasks en 1 thread de OS
proc worker {
    # Ligero, multiplexado
}
```

VENTAJAS:
- Ligero (miles de procs)
- Control total del scheduler
- Alineado con filosofía

DESAVENTAJAS:
- Implementar scheduler complejo
- No usa múltiples cores

---

OPCIÓN 3: Hybrid (Tasks + Worker Pool)
--------------------------------------
```rydit
# Tasks se ejecutan en pool de threads
task worker {
    # Puede correr en cualquier thread
}

# Configurar workers
config workers = 4  # Threads de OS
```

VENTAJAS:
- Usa múltiples cores
- Tasks siguen siendo ligeras
- Control de recursos

DESAVENTAJAS:
- Más complejo
- Thread pools requiere cuidado

---

MI RECOMENDACIÓN: OPCIÓN 3 (Hybrid)

POR QUÉ:
1. **Hardware-first** - Control de threads = control de RAM/CPU
2. **Escalable** - 1 thread en Android, 4+ en desktop
3. **Seguro** - Thread pool limita concurrencia
4. **Educativo** - Enseña concurrencia real

DISEÑO PROPUESTO:
```rust
// En rydit-async o blast-core

pub struct Runtime {
    pool: ThreadPool,  // Fixed-size worker pool
    scheduler: Scheduler,
}

impl Runtime {
    pub fn new(num_threads: usize) -> Self {
        // Crear pool con N threads
    }

    pub fn spawn_task<F>(&self, f: F) -> TaskId
    where
        F: FnOnce() -> Value + Send + 'static,
    {
        // Task se enqueuea al scheduler
        // Scheduler distribuye a workers
    }
}
```

CONFIG RYDIT:
```rydit
# Al inicio del script
config {
    threads = 2,  # Threads de OS
    stack_size = 1024,  # KB por task
}

# Tasks se ejecutan en el runtime
task worker {
    # Código
}
```
