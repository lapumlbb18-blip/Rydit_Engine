## 🔍 Análisis por Feature

### 1. RANDOM - Dificultad: BAJA ⚡

**Complejidad Técnica:** ⭐⭐☆☆☆ (2/5)

**Implementación en Rust:**
```rust
// Opción A: rand crate (estándar)
use rand::Rng;
let mut rng = rand::thread_rng();
let num = rng.gen_range(1..=100);

// Opción B: getrandom + simple PRNG (más ligero)
// Opción C: Usar tiempo como seed (más simple)
```

**Funciones a Implementar:**
| Función | Complejidad | Notas |
|---------|-------------|-------|
| `random::int(min, max)` | Baja | Retorna número entero |
| `random::float()` | Baja | Retorna 0.0-1.0 |
| `random::choice(array)` | Media | Elige elemento de array |
| `random::shuffle(array)` | Media | Mezcla array |

**Dependencias:**
- `rand = "0.8"` → ~500 KB binario
- O sin dependencia: usar `getrandom` + PRNG propio (~100 KB)

**Desafíos en Termux/Android:**
- ✅ `getrandom` funciona en Android
- ⚠️ `rand` puede tener problemas de compilación en ARM
- ✅ Alternativa: PRNG simple (xorshift) sin dependencias

**Tiempo Estimado:** 1-2 sesiones (8-16 horas)

**Veredicto:** ✅ **VIABLE - Baja prioridad de riesgo**

### 2. TIME - Dificultad: MEDIA ⚠️

**Complejidad Técnica:** ⭐⭐⭐☆☆ (3/5)

**Implementación en Rust:**
```rust
// Opción A: chrono crate (completa pero pesada)
use chrono::prelude::*;
let now = Local::now();
let formatted = now.format("%Y-%m-%d %H:%M:%S");

// Opción B: std::time (limitado pero ligero)
use std::time::{SystemTime, UNIX_EPOCH};
let duration = SystemTime::now().duration_since(UNIX_EPOCH)?;

// Opción C: time crate (intermedio)
```

**Funciones a Implementar:**
| Función | Complejidad | Notas |
|---------|-------------|-------|
| `time::now()` | Baja | Timestamp actual |
| `time::sleep(ms)` | Baja | Pausa ejecución |
| `time::format(timestamp)` | Media | Formatear fecha |
| `time::parse(string)` | Alta | Parsear fecha |
| `time::year/month/day()` | Media | Componentes |

**Dependencias:**
- `chrono = "0.4"` → ~1.5 MB binario ⚠️
- `time = "0.3"` → ~800 KB binario
- Sin dependencia: solo `std::time` (limitado)

**Desafíos en Termux/Android:**
- ⚠️ `chrono` tiene dependencias complejas (iana-time-zone)
- ⚠️ Timezones en Android pueden ser problemáticos
- ✅ `std::time` funciona sin problemas
- ⚠️ Formateo de fechas requiere librería externa

**Tiempo Estimado:** 2-3 sesiones (16-24 horas)

**Veredicto:** ⚠️ **VIABLE - Prioridad media de riesgo**

### 3. REGEX - Dificultad: ALTA 🔴

**Complejidad Técnica:** ⭐⭐⭐⭐☆ (4/5)

**Implementación en Rust:**
```rust
// regex crate (estándar de la industria)
use regex::Regex;
let re = Regex::new(r"^\d{3}-\d{2}-\d{4}$").unwrap();
let is_match = re.is_match("123-45-6789");

// Alternativas más ligeras:
// - fancy-regex (más features, más pesado)
// - glob (solo patrones simples)
```

**Funciones a Implementar:**
| Función | Complejidad | Notas |
|---------|-------------|-------|
| `regex::test(pattern, text)` | Media | Retorna bool |
| `regex::match(pattern, text)` | Alta | Retorna matches |
| `regex::replace(pattern, text, repl)` | Alta | Reemplazar |
| `regex::split(pattern, text)` | Media | Dividir string |
| `regex::captures(pattern, text)` | Muy Alta | Grupos de captura |

**Dependencias:**
- `regex = "1.10"` → ~2 MB binario ⚠️⚠️
- Memoria runtime: ~5-10 MB por regex compilado
- Compile time: 30-60 segundos adicionales

**Desafíos en Termux/Android:**
- 🔴 `regex` crate es PESADO (2MB+)
- 🔴 Compile time largo en dispositivos con poca RAM
- ⚠️ Regex complejas consumen mucha memoria
- ⚠️ Posibles OOM (Out Of Memory) en dispositivos < 4GB
- ✅ Alternativa: `glob` para patrones simples (100 KB)

**Problemas Específicos:**
1. **Memoria:** regex compila DFA/NFA que consume RAM
2. **Compile time:** `regex` usa codegen que tarda en ARM
3. **Runtime:** Regex complejas = backtracking exponencial
4. **Error handling:** Reportar errores de regex amigablemente

**Tiempo Estimado:** 3-4 sesiones (24-32 horas)

**Veredicto:** 🔴 **ALTO RIESGO - Considerar alternativas**

## ⚠️ Problemas Potenciales en Termux/Android

### 1. Memoria RAM Limitada

**Escenario:** Dispositivo con 4GB RAM
- Sistema Android usa ~2GB
- Termux disponible: ~1-1.5GB
- `regex` compilando: ~500MB-1GB
- **Riesgo:** OOM killer puede matar el proceso

**Mitigación:**
- Usar `codegen-units = 1` (ya configurado)
- Usar `opt-level = "z"` (ya configurado)
- Considerar swap en SD card (lento)

### 2. Tiempo de Compilación

**Escenario:** Snapdragon 860 (6 cores)
- `regex` crate: 30-60 compilación inicial
- Incremental: 10-15s
- **Impacto:** Desarrollo más lento

**Mitigación:**
- sccache configurado ✅
- Compilar en release solo para benchmarks

### 3. Tamaño del Binario

**Escenario:** Usuario con almacenamiento limitado
- Proyecto actual: ~850 KB
- Con regex: ~4 MB
- **Impacto:** 4x más grande

**Mitigación:**
- Features opcionales en Cargo.toml
- Compilar sin regex por defecto
