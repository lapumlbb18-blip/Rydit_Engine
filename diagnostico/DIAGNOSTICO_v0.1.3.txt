## 📋 Lo que se Encontró

### Bug #1: Parser de Comentarios [CRÍTICO - DESCUBIERTO POR AZAR]

**Descubrimiento:**
Al ejecutar `ejemplo.rydit` para verificar que todo funcionaba, el output mostró:
```
[RYDIT] 9 statements en AST
[ERROR] Solo se puede indexar arrays
```

Pero el archivo tiene 18+ statements. ¿Dónde estaban los otros 9?

**Investigación:**
```bash
# Test mínimo para reproducir
cargo run --bin rydit-rs -- "shield.init # comment dark.slot x = 10 voz x"
# Resultado: 1 statement (debería ser 3)
```

**Causa Raíz (Análisis de Código):**

En `crates/lizer/src/lib.rs`:

```rust
// LÍNEA 672 - Handler de comentarios
Token::Comentario(_) => {
    self.pos += 1;  // ← PROBLEMA #1: Avanza posición
    Ok(None)        // ← Retorna None
}

// LÍNEA 590-595 - Bucle de parse()
while self.pos < self.tokens.len() {
    if let Some(stmt) = self.parse_statement()? {
        statements.push(stmt);
    } else {
        self.pos += 1;  // ← PROBLEMA #2: Avanza OTRA VEZ
    }
}
```

**Resultado:** Doble avance → statement después del comentario se pierde.

**¿Cómo pasó esto?**
- El código fue escrito en la sesión 8 (v0.0.10) - hace 10 sesiones atrás
- Los tests existentes no cubrían comentarios entre statements
- El bug se escondió porque la mayoría de tests no usan comentarios extensivamente
- `ejemplo.rydit` tenía comentarios Y array indexing → bug expuesto

**Fix Aplicado:**
```rust
Token::Comentario(_) => {
    // No avanzar self.pos aquí - el bucle de parse() ya lo hace
    Ok(None)
}
```

**Tests de Regresión:**
```rust
#[test]
fn test_regresion_comentarios_no_saltan_statements() {
    // Bug v0.1.3: Los comentarios hacían que se salte el siguiente statement
    let script = r#"shield.init
# comentario
dark.slot x = 10
voz x"#;
    let tokens = Lizer::new(script).scan();
    let mut parser = Parser::new(tokens);
    let program = parser.parse().unwrap();
    
    assert_eq!(program.statements.len(), 3);  // ANTES: fallaba
}
```

**Veredicto:**
- ✅ **Bug corregido**
- ✅ **Tests de regresión añadidos**
- ⚠️ **¿Cómo pasó 10 sesiones sin detectarse?** - Cobertura de tests insuficiente

### Bug #2: Warnings del Snake Binary [MENOR]

**Descubrimiento:**
```bash
cargo build 2>&1 | grep warning
# warning: field `teclas_presionadas` is never read
# warning: method `es_presionada` is never used
```

**Causa Raíz:**
Código preparado para funcionalidad futura (input handling completo) pero no usado en la implementación actual simplificada.

**Opciones Consideradas:**

| Opción | Pros | Contras | Decisión |
|--------|------|---------|----------|
| Eliminar código | 0 warnings | Perder código útil para futuro | ❌ |
| Implementar feature | Código usado | Scope creep, distrae del fix | ❌ |
| `#[allow(dead_code)]` | Mantiene código, 0 warnings | "Silencia" warnings | ✅ |

**Fix Aplicado:**
```rust
struct InputEstado {
    #[allow(dead_code)]  // ← Para uso futuro
    teclas_presionadas: HashMap<String, bool>,
}

impl InputEstado {
    #[allow(dead_code)]
    fn actualizar(&mut self, _gfx: &RyditGfx) { ... }

    #[allow(dead_code)]
    fn es_presionada(&self, tecla: &str) -> bool { ... }
}
```

**Veredicto:**
- ✅ **Warnings corregidos**
- ✅ **Código futuro preservado**
- ℹ️ **Uso de `#[allow(dead_code)]` es legítimo** - código para features planeados

## 🔍 Análisis Crítico del Bug del Parser

### ¿Por qué es Grave este Bug?

1. **Silencioso:** No hay error, el código "funciona" pero mal
2. **Inconsistente:** A veces funciona (sin comentarios), a veces no
3. **Difícil de debuggear:** ¿Por qué mi script no hace lo que debería?
4. **Se esconde bien:** Tests sin comentarios extensivos no lo detectan

### ¿Cómo Sobrevivió 10 Sesiones?

| Razón | Explicación |
|-------|-------------|
| Tests insuficientes | La mayoría de tests no usan comentarios entre statements |
| Scripts simples | Scripts de prueba eran de una línea o sin comentarios |
| Sesgo de confirmación | "Si compila y no hay error, funciona" |
| Coverage gap | No hay test que diga "comments between statements should work" |

### Lección Aprendida

**Regla para futuros parsers:**
> "Si el bucle principal maneja el avance de tokens, los handlers individuales NUNCA deben avanzar el cursor al retornar `Ok(None)`"

**Acción correctiva:**
- ✅ Tests de regresión añadidos
- ✅ Documentado en lecciones aprendidas
- ⚠️ ¿Deberíamos auditar otros handlers en busca del mismo patrón?
