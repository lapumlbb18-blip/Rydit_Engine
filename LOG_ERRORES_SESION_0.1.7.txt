ERROR #1: COMENTARIOS // NO FUNCIONABAN → CAMBIAR A #
================================================================================

SEVERIDAD: CRÍTICO
IMPACTO: Todos los scripts con comentarios fallaban

DESCRIPCIÓN:
El lexer solo reconocía comentarios con '#' pero la documentación y ejemplos
usaban '//' estilo C/Rust.

ERROR ENCONTRADO EN:
- crates/lizer/src/lib.rs - Lexer scanning
- Múltiples scripts .rydit con // comentarios

SÍNTOMA:
```rydit
// Esto es un comentario
shield.init  // Esto también
# Esto sí funcionaba
```

El lexer tokenizaba `//` como dos tokens `Div` separados, causando:
- Error de parsing
- Statements después del comentario se perdían

SOLUCIÓN APLICADA:
- Estandarizar a `#` para comentarios (Python/Shell style)
- Actualizar todos los ejemplos y demos
- Documentar en README.md

LECCIÓN:
Los comentarios estilo C (//) requieren manejo especial en el lexer.
Para lenguaje scripting, `#` es más apropiado y simple.

ERROR #2: random::int() RETORNABA FLOTANTES EN LUGAR DE ENTEROS
================================================================================

SEVERIDAD: CRÍTICO
IMPACTO: Función random::int() inútil para casos de uso reales

DESCRIPCIÓN:
La función random::int() del módulo random estaba retornando valores f64
en lugar de i64/int, rompiendo la semántica esperada.

ERROR ENCONTRADO EN:
- crates/rydit-rs/src/stdlib/random.rs

SÍNTOMA:
```rydit
import random
dark.slot x = random::int(1, 10)
# x = 7.000000 (flotante) en lugar de 7 (entero)
```

CAUSA RAÍZ:
```rust
// Código incorrecto
pub fn int(min: i64, max: i64) -> f64 {  // ← Tipo de retorno WRONG
    // ... implementación usando PRNG f64
}
```

SOLUCIÓN APLICADA:
```rust
// Código corregido
pub fn int(min: i64, max: i64) -> i64 {  // ← Tipo de retorno CORRECTO
    let range = max - min + 1;
    let random_value = xorshift64() % range as u64;
    min + random_value as i64
}
```

TEST AÑADIDO:
```rust
#[test]
fn test_random_int_returns_integer() {
    let result = int(1, 10);
    assert!(result.is_integer());  // Verifica que es entero
}
```

LECCIÓN:
Los nombres de funciones deben coincidir con su semántica. `int()` debe
retornar enteros, no flotantes.

ERROR #4: json::parse() NO MANEJABA OBJETOS CORRECTAMENTE
================================================================================

SEVERIDAD: CRÍTICO
IMPACTO: Módulo json inútil para objetos JSON reales

DESCRIPCIÓN:
json::parse() fallaba al parsear objetos JSON, solo funcionaba con
arrays y valores primitivos.

ERROR ENCONTRADO EN:
- crates/rydit-rs/src/stdlib/json.rs

SÍNTOMA:
```rydit
import json
dark.slot json_str = '{"nombre": "RyDit", "version": "0.1.7"}'
dark.slot obj = json::parse(json_str)
# ERROR: Failed to parse JSON object
```

CAUSA RAÍZ:
- serde_json::Value parsing no manejaba correctamente la conversión
- Error en el mapeo de objetos JSON a estructura RyDit

SOLUCIÓN APLICADA:
```rust
// Implementación corregida
pub fn parse(json_str: &str) -> Result<Value, RyDitError> {
    let value: serde_json::Value = serde_json::from_str(json_str)
        .map_err(|e| RyDitError::new(format!("JSON parse error: {}", e)))?;
    
    // Manejar todos los casos: Object, Array, String, Number, Bool, Null
    match value {
        serde_json::Value::Object(map) => { /* ... */ }
        serde_json::Value::Array(arr) => { /* ... */ }
        // ... todos los casos
    }
}
```

TEST AÑADIDO:
```rust
#[test]
fn test_json_parse_object() {
    let json = r#"{"key": "value"}"#;
    let result = parse(json);
    assert!(result.is_ok());
}
```

LECCIÓN:
JSON es fundamental para interoperabilidad. El parsing debe ser robusto
y manejar todos los tipos de valores JSON.

ERROR #5: LEXER NO SOPORTABA ESCAPES \" EN STRINGS
================================================================================

SEVERIDAD: MEDIO
IMPACTO: Strings con comillas internas causaban errores

DESCRIPCIÓN:
El lexer no reconocía la secuencia de escape `\"` dentro de strings,
causando que el string terminara prematuramente.

ERROR ENCONTRADO EN:
- crates/lizer/src/lib.rs - Lexer string scanning

SÍNTOMA:
```rydit
dark.slot texto = "dijo "hola""  # ERROR: string sin cerrar
# Debería ser:
dark.slot texto = "dijo \"hola\""  # Pero \\" no funcionaba
```

CAUSA RAÍZ:
```rust
// Código incorrecto - solo avanzaba 1 carácter
if chars[i] == '\\' {
    i += 1;  // ← Solo saltaba el backslash
    // El siguiente carácter se procesaba normalmente
}
```

SOLUCIÓN APLICADA:
```rust
// Código corregido
if chars[i] == '\\' && i + 1 < chars.len() && chars[i + 1] == '"' {
    text.push('"');
    i += 2;  // ← Saltar BOTH: backslash y comilla
    column += 2;
}
```

LECCIÓN:
Los escapes en strings son fundamentales. Cada secuencia de escape debe
consumir BOTH el backslash Y el carácter escapado.
